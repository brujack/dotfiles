# Retrospective — 2026-06-05

**Period:** 2026-05-05 to 2026-06-05
**Repo(s):** dotfiles
**PRs merged:** 44 (PRs #76–#119)
**Direct master commits:** ~65 (docs, chore, memory — mostly post-PR follow-ups)

> Note: A mid-period retrospective (`2026-06-retrospective.md`) covered May 17–June 1. This document covers the full 30-day window and supersedes it for trend analysis.

## Summary

Heaviest-volume month on record. Two dominant themes: (1) a systematic **coverage sprint** (PRs #94–#118) driving bash coverage from ~82% to 92% with 590→729 BATS tests, and (2) **structural cleanup** removing dead platform support (RHEL/CentOS/Fedora, Ubuntu <24.04, powerlevel10k). Feature work was present but secondary. The CI gate is fully automated with 0.0h median cycle time. The main friction is documentation maintenance overhead — CLAUDE.md was touched 58 times and `docs/superpowers/README.md` 38 times in 30 days.

---

## What got easier

- **Auto-merge is zero-friction.** Median cycle time 0.0h — 44 PRs opened and merged in the same session. The 3-job gate (test, lint-macos, bash-coverage) is reliable.
- **Coverage now gated in CI at 90%.** PR #116 replaced the broken kcov approach with PS4 xtrace on ubuntu-latest. Coverage is measured, enforced, and not advisory.
- **Weekly digests run autonomously.** `whats-new-claude-code.sh` and `whats-new-anthropic.sh` both produced Monday digests without intervention.
- **Dead platform support removed without drama.** Removing RHEL/CentOS/Fedora (#89), Ubuntu <18.04–22.04 (#90), and powerlevel10k (#99) trimmed significant conditional complexity. No regressions.
- **Bash coverage measurement is solved.** PS4 xtrace approach documented in ADR-0008. Per-file ceilings documented in CLAUDE.md to prevent wasted effort on structurally untraceable lines.
- **Skills toolbox grew.** `docs`, `release`, and a strengthened `learnings` skill landed in ai-config. Each session now has a clear post-merge closeout path.

## Recurring gotchas (action required)

- **`export -f` required for BATS subshell stubs** (PR #111). A test file was silently not exercising its stubs — the real binary ran, tests passed vacuously. Fixed and documented in CLAUDE.md. This pattern is a consistent blind spot: PATH mocking works for direct calls but not for subprocess-spawning functions. **→ Proposed action:** Add to the pre-review checklist: for any test using a mock for a function that spawns subprocesses, verify `export -f mockname` is present.

- **Coverage PRs too granular** — 8 PRs merged on May 28, each covering one function group. Already noted in the June 1 retro. **→ Proposed action:** One PR per lib file for future coverage sprints. The May 28 burst is the anti-pattern to avoid.

- **`zshrc.d` re-source crash on `readonly` vars** — `[[ -v VAR ]]` guard required; found during zshrc work. Reverted once (#685d23f after #662565c) due to live-shell not tested before merge. **→ Proposed action:** Document `zsh -i -c 'exit'` smoke-test in CLAUDE.md zshrc section. Not yet done.

- **`_UPDATE_SECTION_ORDER` coupling** — adding a new `run_update` section without adding to the array causes silent omission from the printed summary. Documented in CLAUDE.md ("Key Conventions"). **→ Already documented**, no further action needed.

- **`run_update` pip hang with real python3** — without `MOCK_PYENV_WHICH_STDOUT`, generic `run_update` tests hit real pip. Documented in CLAUDE.md Testing Rules. **→ Already documented.**

## Friction points

- **CLAUDE.md 58 touches** — this file is a documentation dump accumulating every discovered gotcha, convention update, and test seam. Now 400+ lines in the project file alone. **→ Proposed action:** Audit for sections that are stable enough to move to a per-topic knowledge doc (e.g. mock pattern table → `docs/knowledge/mock-patterns.md`). The `@include` approach for standards is reducing drift; same pattern could apply to Testing Rules.

- **`docs/superpowers/README.md` 38 touches** — every PR requires a manual status update. Most edits are 1-line changes (status: Done, test count N). **→ Proposed action:** Accept the overhead as the cost of the plan index, but batch the updates: one doc commit per session, not one per PR.

- **Direct master doc commits inflate git log** — 65 direct commits in 30 days, mostly `docs:` and `chore:` followups to PRs. Pattern is intentional (docs and memory can go direct) but makes `git log` noisy. No change proposed — this is a known trade-off.

- **Test health artifacts unavailable for etch-cli / math** — `test_health.py` reports no artifacts because CI doesn't upload test timing data for those repos. Will resolve when Plans 1+2 for those repos merge.
